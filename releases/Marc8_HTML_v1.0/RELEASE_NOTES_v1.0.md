## Purpose

Official Release 1 of the Marc8 Fleet Financial ERP static HTML frontend. This release contains the complete conversion of the original React/Vite frontend into a pure static HTML/CSS/Vanilla JS frontend. The frontend is ready for backend API integration by another developer.

No build step. No server required. No Node.js dependencies. Every page opens by double-clicking `index.html` (or any page directly).

## Known Limitations

### Low Severity

1. **Empty resource directories** — `assets/`, `fonts/`, `icons/`, `images/` are empty placeholders. No runtime impact. Ready for future static assets.

2. **Inline script duplication** — 9 master data pages (customers, vendors, drivers, vehicles, vehicle-owners, vehicle-status, ownership-types, transmission-types, outstanding-categories) define their own `showToast`, `toggleSidebar`, `openCommandPalette` functions inline, duplicating what exists in `app.js`. Works correctly at runtime. May be consolidated in a future release.

### Runtime

3. **No backend** — All data is mock data generated by `data.js` with per-session caching. Data resets on page refresh (dashboard, bookings, etc.) or persists only in memory during the session.

4. **No form submission** — Form submits are handled client-side only (toast + reset). No data is persisted to a server.

5. **No offline support** — Google Fonts requires internet connectivity. All other assets are local.

## Integration Instructions for Backend Developers

### Getting Started

1. **Unzip** `Marc8_HTML_v1.0.zip`
2. **Open any HTML file** — the frontend works by opening files directly (file:// protocol) or via any HTTP server
3. **Login** — click "Continue as Guest" on `login.html` to access the dashboard

### How to Connect a Backend

#### Step 1: Replace Mock Data
- **File:** `js/data.js`
- The `DATA` object (lines 446-466) contains all mock data getters
- Each getter (e.g., `getBookings`, `getExpenses`, `getDashboard`) returns an array or object
- **Replace each getter** with an `async` function that calls your API endpoint
- The `cached()` wrapper on line 5 can be removed when using real APIs

#### Step 2: Wire Up Forms
- **File:** `js/app.js` and inline scripts in individual HTML pages
- Form submissions use `onsubmit` or `onclick` handlers that currently call `showToast()`
- **Replace** the handler body with your API call (e.g., `fetch('/api/endpoint', { method: 'POST', body: JSON.stringify(data) })`)
- All forms already collect data via `FormData` or direct DOM access

#### Step 3: Enable Real Authentication
- **File:** `html/login.html`
- The "Continue as Guest" button calls `handleGuestLogin()` which redirects to `dashboard.html`
- **Replace** with your auth flow (e.g., `POST /api/auth/login` with username/password)
- A real session token should be stored and passed to subsequent API calls

#### Step 4: Serve with an HTTP Server
- The frontend is static. Serve it via:
  - Any HTTP server (nginx, Apache, Python `http.server`, Node `serve`, etc.)
  - A backend framework's static file middleware (Express, Django, Flask, Spring Boot, etc.)
- Ensure all routes that don't match a static file serve `index.html` or return 404 appropriately

### Key Files for Backend Developers

| File | Purpose |
|------|---------|
| `js/data.js` | Mock data layer — replace with API calls |
| `js/app.js` | All frontend logic — see page-specific `init*()` functions |
| `html/*.html` | Every page has `data-page` attribute for app.js routing |
| `css/brand.css` | CSS custom properties for branding/theming |

### Technical Details

- All pages use `<script src="../js/data.js"></script>` before `<script src="../js/app.js"></script>` — this load order must be preserved
- The `window.DATA` global object exposes all data getters to `app.js`
- Page identity is set via `<body data-page="page-name">` — used by `app.js` to initialize page-specific features
- No router — each HTML file is a standalone page. Navigation is via `<a href="...">` links
- All styling uses CSS custom properties defined in `brand.css` — easy to rebrand
